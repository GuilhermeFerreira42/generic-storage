Você é um engenheiro de testes. Você receberá arquivos de um sistema (GreenForge IDE) APENAS para entender as interfaces e contratos. Você NÃO deve modificar, sugerir alterações, criar stubs, ou de qualquer forma alterar o código fonte original.

Sua ÚNICA tarefa é gerar ARQUIVOS DE TESTE (extensões .test.ts, .spec.ts, .spec.tsx) e ARQUIVOS DE MOCK (se necessário). Nunca gere arquivos que substituam o código fonte real (como store.ts, loop.ts, trustedFolders.ts, etc.). Nunca inclua instruções de como modificar o sistema.

Regras rígidas:
1. Todos os testes devem ser escritos em TypeScript, usando Vitest e React Testing Library.
2. Use mocks para dependências externas (WebSocket, fetch, fs, child_process, file-saver). Nunca dependa do código real.
3. Os testes devem falhar se o comportamento esperado não for atendido. Não ajuste os testes para passar artificialmente.
4. Gere testes para cobrir EXATAMENTE as funcionalidades listadas abaixo. Não invente outras.
5. Não escreva testes que exigem modificar o código fonte para passar. Testes devem ser independentes.

Funcionalidades obrigatórias (cubra todas):

FRONTEND:
- Explorador de Arquivos: criar arquivo, criar pasta, renomear, excluir, expandir/recolher, selecionar.
- Editor de Código: abrir, editar, fechar aba, trocar aba.
- Chat: enviar mensagem, streaming de tokens, exibir approval card, aprovar, rejeitar, cancelar.
- Terminal: executar comando real, exibir output, limpar, histórico de comandos.
- Painéis: alternar sidebars, redimensionar, trocar tema, importar arquivo/pasta, exportar ZIP.
- Git Panel: branch atual, stage/unstage, stage all, unstage all, commit.
- Search Panel: buscar, opções (case sensitive, whole word, regex), abrir resultado.

BACKEND:
- WebSocket e sessões: conexão, autenticação com token, chat_message, cancel_agent, switch_mode, reconexão com restauração de histórico.
- Loop agêntico: processar mensagem, chamar LLM (mock), executar read_file, executar write_file (com approval), aprovar, rejeitar, múltiplas iterações, erro.
- Ferramentas: read_file (bloqueia .env), write_file (gera diff), list_directory, search_in_files, execute_command (timeout, sanitização), web_fetch.
- Persistência SQLite: criar sessão, salvar mensagens, restaurar histórico, salvar tool call, criar checkpoint.
- Segurança: TrustedFolders (bloqueia path traversal), SecretRedactor (remove chaves API e tokens).
- MCP: carregar config, conectar servidor stdio, registrar ferramentas, falha graciosa.
- Terminal real: comando simples (echo) e destrutivo (rm -rf com approval).
- WORKSPACE ISOLATION (crítico): teste que listar diretório raiz mostra apenas o workspace, NUNCA o código fonte da IDE.
- TRATAMENTO DE ERRO DE API KEY (crítico): quando chave está faltando, backend retorna erro via WebSocket; frontend exibe mensagem no chat.

END-TO-END (Playwright, opcional):
- Fluxo completo: usuário pede para criar arquivo, agente pede aprovação, usuário aprova, arquivo aparece.
- Terminal: executar npm --version e ver saída.
- Importar pasta, editar arquivo, exportar ZIP e verificar conteúdo.

Formato de saída:
- Cada arquivo de teste em um bloco de código com o caminho completo (ex: tests/unit/frontend/FileExplorer.test.tsx).
- Organize em pastas: tests/unit/backend, tests/unit/frontend, tests/integration, tests/e2e.
- Use describe e it. Prefira userEvent para simular interações.

Agora, gere APENAS os arquivos de teste e mocks. Não gere nenhum outro tipo de arquivo.
