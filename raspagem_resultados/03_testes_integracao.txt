## 2. MOCKS DO GEMINI CLI DOCUMENTADOS

### MockExtensionContext (GREENFORGE_DESIGN.md - Seção de Testes)
```typescript
class MockExtensionContext implements ExtensionContext {
  extensionPath = '/mock/greenforge';
  globalState = new MockMemento();
  workspaceState = new MockMemento();
  subscriptions: Disposable[] = [];
  log = new MockLogger();
  
  dispose() {
    this.subscriptions.forEach(s => s.dispose());
  }
}
```

### MockMemento (para globalState/workspaceState)
```typescript
class MockMemento implements Memento {
  private store = new Map<string, any>();
  
  get<T>(key: string): T | undefined {
    return this.store.get(key);
  }
  
  set(key: string, value: any): void {
    this.store.set(key, value);
  }
}
```

### MockToolRegistry
```typescript
class MockToolRegistry {
  registeredTools = new Map<string, ToolDefinition>();
  
  registerTool(name: string, schema: ToolSchema, handler: ToolHandler) {
    this.registeredTools.set(name, { name, schema, handler });
  }
  
  async executeTool(name: string, args: any) {
    const tool = this.registeredTools.get(name);
    if (!tool) throw new Error(`Tool ${name} not found`);
    return tool.handler(args);
  }
}
```

## 3. CENÁRIOS DE TESTE DE INTEGRAÇÃO (Gherkin)

### Cenário 1: Ativação da Extensão
```gherkin
Dado que o Gemini CLI está inicializando
Quando a extensão GreenForge é ativada
Então activate() deve ser chamado com ExtensionContext válido
E StateMachine deve estar no estado INITIALIZED
E Database SQLite deve estar conectada
E worktreeRoot deve estar configurado
```

### Cenário 2: Registro de Ferramentas
```gherkin
Dado que a extensão foi ativada
Quando registerTool() é chamado para forge_start_task
E registerTool() é chamado para forge_list_tasks
E registerTool() é chamado para forge_approve_plan
Então todas as 3 ferramentas devem estar registradas
E o schema de cada ferramenta deve ser válido
```

### Cenário 3: Interceptação de Mensagem
```gherkin
Dado que onMessage hook está registrado
Quando usuário envia "/greenforge-start nova-tarefa"
Então onMessage deve interceptar antes do Gemini CLI processar
E NEXUS deve parsear o comando
E StateMachine deve transitar para INITIALIZED
```

### Cenário 4: Execução de Ferramenta
```graherkin
Dado que uma tarefa está ativa
Quando usuário invoca forge_approve_plan
Então onToolCall deve validar os parâmetros
E GateManager deve aprovar o plano
E DiffLens deve aplicar as mudanças
E StateMachine deve atualizar estado
```

### Cenário 5: Persistência de Estado
```gherkin
Dado que uma tarefa está em execução
Quando workspaceState.set() é chamado
Então o estado deve ser persistido no SQLite
E após restart, workspaceState.get() deve retornar valor correto
```

### Cenário 6: Desativação Limpa
```gherkin
Dado que há tarefas ativas
Quando deactivate() é chamado
Então todas as tarefas devem ser salvas
E recursos devem ser liberados
E subscriptions devem ser disposed
```

### Cenário 7: Error Handling
```gherkin
Dado que Extension API lança erro
Quando onToolCall é executado
Então erro deve ser capturado e logado
E usuário deve receber mensagem amigável
E StateMachine não deve corromper
```
